\section{Introduction}

\subsection{Purpose}
\hspace{-0.8cm}
The \textbf{Students\&Companies (S\&C)} platform aims to simplify the process of connecting university students with companies offering internships. For students, finding internships that match their skills and career goals can often be challenging, while companies may struggle to identify candidates with the right qualifications.

\hspace{-0.8cm}
This platform provides students with tools to search for internships, receive personalized recommendations, and manage their applications. At the same time, companies can use it to advertise internships, review student CVs, and organize interviews efficiently. By incorporating advanced recommendation mechanisms, the platform creates a streamlined and mutually beneficial experience for both parties.

\subsection{Scope}
\hspace{-0.8cm}
The S\&C platform is designed for two main types of users: students and companies. 

\begin{itemize}
    \item \textbf{Students}: Students can create and upload their CVs, search for internships, receive personalized recommendations, and track their application progress.
    \item \textbf{Companies}: Companies can advertise internships, search for and evaluate student profiles, schedule interviews, and manage the selection process.
\end{itemize}

\hspace{-0.8cm}
Additionally, the platform gathers feedback from both students and companies. This feedback is used to enhance the recommendation system and provide insights into the effectiveness of internships, ensuring continuous improvement of the platform.

\section{Architectural Design}

\subsection{Overview}

    \hspace{-0.8cm}
    In this section are briefly presented the main design choices of the S\&C platform architecture.

    \begin{itemize}
        \item \textbf{Microservices}: This architectural style divides the functionalities of the system in smaller services. This improves the scalability, maintainability and availability of the system, also reducing dependencies between modules. A detailed description of such microservices will be provided in the following sections.
        \item \textbf{REST API}: This is the main API which enables the communication between different components, for example between server side and client side, and between different microservices. Each microservice will expose its operation through the \textit{REST API}.
    \end{itemize}

\subsection{Component View}

    \hspace{-0.8cm}
    The following diagram shows the main components of the system and the interfaces through which they communicate.
    The system is composed of two sides:

    \begin{itemize}
        \item The \textbf{Client side} is composed of two components, the Student and Company web app.
        \item The \textbf{Server side} is composed of several components that offer different functionalities. In the following diagram they are represented as standalone, with a brief description in the following part, while, in section \textit{2.3}, it is shown how they are organized in several microservices.
    \end{itemize}

    \begin{figure}[H]
        \centering
        \hspace*{-1.2cm}
        \includegraphics[scale = 0.5]{DD/dd_diagrams/ComponentDiagram_v_1_3.png}
        \caption{Component Diagram}

    The main components of the system are:
    \begin{itemize}
        \item \textbf{Web App}: The web app (\textless\textless Student \textgreater\textgreater, \textless\textless Company\textgreater\textgreater) represents the way in which the users can interact with the platform.
        \item \textbf{StudentAuthenticator \& CompanyAuthenticator}: These components handle the authentication process (e.g. login, registration) for, respectively, student and companies.
        \item \textbf{CompanyProfileManager \& StudentProfileManager}: These components work together with the \textit{Authenticators} components in the authentication process. They directly interact with the student or company DBMS to check eventual inconsistencies in the credentials, such as wrong password or already existing username. Moreover these components are useful to retrieve the students or companies information, which can be used, for example, to schedule an interview, or to notify a student about an internship he could apply to.
        \item \textbf{CVManager}: This component is used by the student to upload or modify his CV.
        \item \textbf{InternshipManager}: This component is used by the company to manage its internships.
        \item \textbf{SearchEngine}: This component is used by the student to search for internships. This engine uses the information it retrieves from the \textit{InternshipManager} to find the proper internships according to the constraints set by the student.
        \item \textbf{ApplicationManager}: This component is used by both the student and the company. The student can apply to an internship and view the state of his application. The company, on the other hand, can assess the applications it receives.
        \item \textbf{InterviewManager}: This component is used by both the student and the company. The company can schedule an interview, send the questionnaires to the student and check his answers. On the other hand, the student receives the questionnaires and answers them.
        \item \textbf{FeedbackManager}: This component is used to ask for feedback to both the company and the student. This feedback will be used by the \textit{Recommendation System} to search for any suitable match between students and internships.
        \item \textbf{RecommendationSystem}: This component periodically retrieves all the information about internship, CVs and feedback, and uses them to search for suitable matches between a student and an internship.
        \item \textbf{NotificationManager}: This component is responsible for handling the platform notifications both for students and companies, such as the assessment of a student's application or a found match between a student and a company.
        \item \textbf{DBMS}: The DBMS components are used to store the state of the system and its evolution, also allowing the user to retrieve information received while disconnected.
    \end{itemize}

    \hspace{-0.8cm}
    \textbf{Connection to the server} \\
    Student and companies can access the S\&C platform through their smartphones or computers thanks to the web app.
    Before arriving at the application server, the request must pass through a \textit{load balancer} and an \textit{API gateway}.
    The first redirects the request to one of the instances of the gateway, which main characteristics are:
    \begin{itemize}
        \item \textbf{Routing}: Each request is redirected to the right service, chosen based on the destination address of the request.
        \item \textbf{Security}: The client speaks only to the gateway, which implements the authentication and authorization mechanism.
    \end{itemize}
    \vspace{0.5cm}

    \hspace{-0.8cm}
    \textbf{Microservices} \\
    Figure \ref{fig:deployment_diagram} shows how the system is divided into 9 different services.
    Each of them has its own personal database, this may cause higher coupling, in fact, services need to interact with each other, but, on the other hand, only one entity can modify each database and, if a DBMS interface needs to undergo any change, only a small number of components would be modified. \\
    The microservice architectural style choice guarantees to improve the scalability, maintainability, and ease of deployment of the system.

\subsection{Selected Architectural Styles and Patterns}

\subsubsection{Architectural Styles}
    \hspace{-0.8cm}
    The microservices architecture is designed to improve the maintainability, resilience, availability, and scalability of the system through selective replication. In addition, it allows multiple teams to work on different microservices in parallel. \\
    The use of a \textit{REST API} facilitates communication between various services, which requires the use of HTTPS and standard HTTP methods to perform CRUD operations such as GET, POST, PUT, and DELETE

\subsubsection{Security Pattern}
    \hspace{-0.8cm}
    The use of an \textit{API Gateway} is a key element for both routing and security purposes. This gateway acts as an intermediary between the client and the server side of the application, meaning that the client communicates exclusively with the gateway, which is responsible for mapping requests to the appropriate microservice. Although the API Gateway could represent a single point of failure, this limitation can be mitigated by deploying multiple replicas of the component and implementing a load balancer to distribute incoming requests.

\section{Implementation, Integration and Test Plan}
\hspace{-0.8cm}
This chapter explains how to implement, integrate, and test the system designed in this
document. As already stated and described in the previous sections of this document, the
architectural style employed for the development of the S\&C system is represented by the
microservices. Among the other advantages of this design decision, a key aspect of this choice is that it allows the system to be
developed by several different sets working on different microservices in parallel. To
illustrate the test plan and the integration, we will refer to the respective teams as Set A, Set B, Set C, Set D, Set E, Set F, Set G. Following the just mentioned details, it can be
inferred that the employed approach is a parallel bottom-up approach, in which each team
focuses on its own microservices and, after their implementation, their integration is tested.
